# Проектирование базы данных

## Назначение модели

Схема БД покрывает MVP дипломного проекта для бронирования, продажи и возврата железнодорожных билетов. Модель опирается на MySQL `InnoDB`, внешние ключи, уникальные индексы и проверки на уровне таблиц, а критические операции бронирования и оплаты предполагают выполнение внутри транзакций backend.

## Ключевые сущности

- `roles` -> справочник ролей `passenger`, `cashier`, `admin`.
- `users` -> учётные записи пользователей с привязкой к роли и хэшем пароля.
- `stations` -> станции отправления и прибытия.
- `routes` -> маршрут как направление между конечными станциями.
- `route_stations` -> состав маршрута по станциям и порядку остановок.
- `trains` -> поезда.
- `carriages` -> вагоны внутри конкретного поезда.
- `seats` -> места внутри вагона.
- `trips` -> конкретный рейс по маршруту и поезду на дату и время.
- `bookings` -> центральная таблица жизненного цикла места на рейсе.
- `tickets` -> билет как выдаваемый артефакт после успешной оплаты.
- `payments` -> записи об имитации оплаты или оплате через кассира.
- `refunds` -> возвраты по оплаченным билетам.
- `operation_logs` -> аудит действий пользователей и служебных операций.

## Логика жизненного цикла

`bookings` хранит бизнес-статус места на рейсе:

- `reserved` -> бронь активна, место временно занято, обязателен `reserved_until`.
- `paid` -> место окончательно продано, обязателен `paid_at`.
- `cancelled` -> бронь отменена вручную.
- `expired` -> бронь истекла по времени.
- `refunded` -> оплаченный билет возвращён, обязателен `refunded_at`.

`tickets` не дублирует всю бизнес-логику брони. Эта таблица хранит номер билета, время выпуска и собственный технический статус артефакта: `issued`, `refunded`, `void`.

## Связи ER-модели

- Один `role` имеет много `users`.
- Одна `station` может участвовать во многих `routes` и `route_stations`.
- Один `route` имеет много `route_stations` и много `trips`.
- Один `train` имеет много `carriages` и много `trips`.
- Один `carriage` имеет много `seats`.
- Один `trip` имеет много `bookings`.
- Один `booking` принадлежит одному `user`, одному `trip`, одному `carriage` и одному `seat`.
- Один `booking` может иметь максимум один `ticket`.
- Один `booking` может иметь несколько `payments`, но только один итоговый `ticket`.
- Один `ticket` и один `booking` могут иметь максимум один `refund`.
- `operation_logs` могут ссылаться на пользователя-инициатора и логируемую сущность.

## Защита от двойной продажи

Главный механизм находится в `bookings`:

- введён вычисляемый столбец `active_trip_seat_key`;
- он заполняется только для статусов `reserved` и `paid`;
- на него поставлен уникальный индекс `uq_bookings_active_trip_seat`.

Это означает, что в БД не может существовать две одновременно активные записи для одной комбинации `trip_id + carriage_id + seat_id`.

Практический сценарий работы:

- при бронировании backend открывает транзакцию;
- повторно проверяет место по `trip_id`, `carriage_id`, `seat_id`;
- вставляет запись в `bookings`;
- при конфликте уникального индекса операция отклоняется;
- при оплате та же запись переводится из `reserved` в `paid`, а не создаётся новая.

## Ограничения и индексы

В схеме добавлены:

- уникальные ключи на логины, email, коды станций, коды маршрутов, номера билетов, ссылки платежей и возвратов;
- составные индексы для частых запросов поиска рейсов, мест и броней;
- внешние ключи для защиты ссылочной целостности;
- `CHECK`-ограничения для дат, статусов и положительных числовых значений;
- отдельный индекс по `reserved_until` для фонового поиска просроченных броней.

## Важные допущения и ограничения

1. MySQL не умеет выразить правило "бронь активна только пока `reserved_until > NOW()`" через частичный уникальный индекс. Поэтому повторное использование места после истечения брони требует перевода записи из `reserved` в `expired` или `cancelled` фоновым процессом backend.
2. Согласованность между `carriage_id` и `seat_id` внутри `bookings` должна дополнительно проверяться приложением. Внешние ключи подтверждают существование вагона и места, но не проверяют, что место принадлежит именно указанному вагону в рамках одного ограничения.
3. Прямая продажа кассиром поддерживается через создание записи в `bookings` сразу со статусом `paid` и последующим выпуском `ticket`.
4. Для MVP не вводилась отдельная таблица истории смены статусов. История восстанавливается по `bookings`, `payments`, `refunds` и `operation_logs`. Если позже потребуется полный аудит переходов, можно добавить `booking_status_history` отдельной миграцией.
5. Сессии авторизации в эту схему не включены, потому что задача текущего этапа ограничена предметной БД билетов. Хранилище server-side sessions можно добавить отдельно при реализации auth.
