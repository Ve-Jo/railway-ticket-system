# Архітектура проєкту

## Загальна структура

Проєкт реалізовано як монорепозиторій із поділом на клієнтську, серверну та SQL-частину.

- `client` - користувацький інтерфейс на React + Vite
- `server` - REST API на Node.js + Express
- `db` - схема бази даних, тестові дані та SQL-перевірки
- `docs` - проєктна документація
- `scripts` - допоміжні сценарії ініціалізації та супроводу

## Архітектурний підхід

Для дипломного MVP обрано просту прикладну архітектуру:

- frontend і backend зберігаються в одному репозиторії;
- сервер надає HTTP API для всіх клієнтських сценаріїв;
- клієнт працює поверх API і не звертається до БД напряму;
- автентифікація побудована на серверних сесіях і cookie;
- MySQL використовується як основне джерело істини для предметних даних.

Такий формат зручний для демонстрації, розгортання та подальшої підтримки.

## Основні серверні модулі

- `auth` - реєстрація, вхід, вихід, поточна сесія;
- `stations` - довідник станцій;
- `routes` - маршрути та їхні проміжні зупинки;
- `trips` - рейси на конкретну дату;
- `trains` - поїзди, вагони, місця;
- `bookings` - бронювання місць;
- `payments` - імітація оплати;
- `tickets` - видані квитки;
- `refunds` - повернення квитків;
- `admin` - адміністративні CRUD-сценарії;
- `logs` - журналювання важливих операцій.

## Пріоритет розробки

Робота над системою йде не за кількістю сторінок, а за ризиком і бізнес-цінністю.

### Перша черга

- схема БД;
- підключення сервера до MySQL;
- сесії та рольовий доступ;
- пошук рейсів;
- перегляд місць;
- захист від подвійного бронювання;
- переведення броні в оплачений квиток.

### Друга черга

- повернення квитків;
- службовий інтерфейс касира;
- адміністративні CRUD-модулі;
- журнал операцій.

### Третя черга

- полірування UX;
- валідації та повідомлення про помилки;
- розширені демо-дані;
- покращення візуальної частини.

## Ключові бізнес-правила

- одне місце на конкретному рейсі не може бути одночасно заброньоване або продане більше одного разу;
- сервер повторно перевіряє доступність місця перед записом у БД;
- бронювання має строк дії;
- оплачений квиток має окремий життєвий цикл від бронювання;
- повернення дозволене лише для оплаченого квитка до моменту відправлення рейсу.

## Основні групи API

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `GET /api/stations`
- `GET /api/trips/search`
- `GET /api/trips/:id`
- `POST /api/bookings`
- `POST /api/bookings/:id/cancel`
- `POST /api/payments`
- `GET /api/tickets/my`
- `POST /api/refunds`
- `GET /api/admin/...`

## Перший завершений цикл

Базовий етап вважається завершеним, якщо в репозиторії є:

- зафіксована схема БД;
- опис архітектурних рішень;
- стартовий каркас frontend і backend;
- інструкції локального запуску;
- мінімальний пасажирський сценарій від пошуку рейсу до оплати квитка.
