# Архитектура проекта

## Общая структура

Проект реализуется как веб-приложение с разделением на клиентскую и серверную части.

- `client` — пользовательский интерфейс на React + Vite
- `server` — REST API на Node.js + Express.js
- `db` — SQL-схема и связанные с базой материалы
- `docs` — проектная документация

## Архитектурный подход

Для дипломного MVP выбирается простой монолитный подход:

- frontend и backend находятся в одном репозитории;
- backend предоставляет HTTP API;
- frontend обращается к backend по API;
- аутентификация строится через серверные сессии и cookie;
- MySQL используется как основное хранилище данных.

Такой формат проще в запуске, демонстрации и сопровождении.

## Основные backend-модули

Серверная часть должна быть разделена на простые прикладные модули:

- `auth` — регистрация, вход, выход, текущая сессия;
- `users` — профиль пользователя и роли;
- `stations` — станции;
- `routes` — маршруты и точки маршрута;
- `trips` — рейсы и расписание;
- `trains` — поезда, вагоны, места;
- `bookings` — бронирования;
- `tickets` — оплаченные билеты;
- `payments` — имитация оплаты;
- `refunds` — возвраты;
- `admin` — административные операции;
- `logs` — журналирование действий.

## Приоритет разработки

Разработка должна идти не по числу экранов, а по риску и по бизнес-ценности.

### Первая очередь

- схема базы данных;
- сервер и подключение к MySQL;
- сессии и разграничение доступа;
- поиск рейсов;
- выбор места;
- защита от двойного бронирования;
- перевод брони в оплаченный билет.

### Вторая очередь

- возвраты;
- интерфейс кассира;
- административные CRUD-модули;
- журнал операций.

### Третья очередь

- улучшение UX;
- валидации и сообщения об ошибках;
- демонстрационные данные;
- полировка интерфейса.

## Ключевые бизнес-правила

- место на конкретном рейсе не может быть одновременно продано или активно забронировано более одного раза;
- сервер повторно проверяет доступность места перед записью;
- бронь имеет срок действия;
- оплаченный билет имеет отдельный жизненный цикл от брони;
- возврат применим только к оплаченному билету до момента отправления.

## Основные API-группы

Предварительно API можно разделить на такие группы:

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `GET /api/stations`
- `GET /api/trips/search`
- `GET /api/trips/:id`
- `POST /api/bookings`
- `POST /api/bookings/:id/cancel`
- `POST /api/payments`
- `GET /api/tickets/my`
- `POST /api/refunds`
- `GET /api/admin/...`

Точные маршруты могут быть уточнены при реализации, но логическое деление следует сохранить.

## Первый рабочий цикл

Первый цикл разработки должен завершиться следующими артефактами:

- зафиксированная схема базы данных;
- документ с архитектурными решениями;
- стартовый каркас frontend и backend;
- инструкции по запуску проекта локально.

После этого можно переходить к реализации аутентификации и базовых API.
